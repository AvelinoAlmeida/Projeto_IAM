\section{Arquitetura}

A solução segue o padrão \textbf{Resource Server + Identity Provider}, com três serviços orquestrados por Docker Compose (Figura~\ref{fig:arquitetura}). Toda a configuração é declarativa: \texttt{docker-compose.yml}, \texttt{realm-export.json} (versionado em git) e \texttt{.env} (segredos, não versionado).

\begin{figure}[h!]
  \centering
  \includegraphics[width=0.92\textwidth]{arquitetura_iam.png}
  \caption{Arquitetura: FastAPI (\textit{resource server}), Keycloak 26 (\gls{idp}) e PostgreSQL 16, sob Docker Compose.}
  \label{fig:arquitetura}
\end{figure}

\textbf{Fluxo principal}: cliente autentica no Keycloak → recebe \gls{jwt} RS256 → apresenta à FastAPI → validação local via \gls{jwks} cacheado → \gls{rbac} aplicado → resposta.

\begin{lstlisting}[caption={Arranque completo da infraestrutura},label={lst:docker}]
docker compose up --build
curl http://localhost:8000/health  # {"status":"ok"}
\end{lstlisting}

\section{Fluxos de Autenticação e Autorização}

\subsection{Validação JWT (decisão técnica)}

A validação é feita \textbf{localmente} na FastAPI com a chave pública RS256 obtida via \gls{jwks} e cacheada em memória. Verifica-se assinatura, \texttt{issuer}, \texttt{audience} e expiração. As \textit{claims} extraídas são: \texttt{sub}, \texttt{email}, \texttt{preferred\_username}, \texttt{realm\_access.roles} e \texttt{amr}.

\begin{lstlisting}[language=Python, caption={Validação JWT em auth.py},label={lst:jwt}]
payload = jwt.decode(
    token, jwks,
    algorithms=["RS256"],
    audience=settings.oidc_client_id,
    options={"verify_exp": True},
)
\end{lstlisting}

\section{MFA / TOTP (decisão técnica)}

O utilizador \texttt{admin.user} tem a \textit{Required Action} \texttt{CONFIGURE\_TOTP}: no primeiro acesso via \textit{browser}, o Keycloak obriga a configurar uma aplicação TOTP. Após configuração, o \gls{jwt} inclui \texttt{"amr":["otp"]}. O endpoint \texttt{/admin/mfa-area} valida este \textit{claim} com \texttt{require\_mfa()} independentemente do Keycloak — token sem \gls{otp} retorna 403.

\section{Implementação JML}

Os três fluxos actuam sobre a Keycloak Admin \gls{rest} \gls{api} (Figura~\ref{fig:jml}). A \textbf{decisão técnica} central é a revogação explícita de sessões (\texttt{DELETE /users/\{id\}/sessions}) no Mover e Leaver, garantindo efeito imediato sem aguardar a expiração do token (300 s por omissão).

\section{Auditoria e Riscos}

O Keycloak regista eventos de utilizador (logins, falhas, logout, \gls{totp}) e eventos de administração (\gls{jml}) com persistência em PostgreSQL. O endpoint \texttt{GET /admin/audit} (papel \texttt{admin}) agrega ambos com filtragem por tipo.

\textbf{Excerto de evento de login} (retornado pelo \texttt{/admin/audit}):

\begin{lstlisting}[caption={Excerto de evento de auditoria — login bem-sucedido},label={lst:audit}]

  "clientId": "fastapi-client",
  "userId": "a1b2c3d4-...",
  "ipAddress": "172.18.0.1",
  "details": { "username": "colaborador.user" }
}
\end{lstlisting}
