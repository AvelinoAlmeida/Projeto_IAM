1. Objetivo e Planeamento

O objetivo do projeto é construir uma solução IAM completa para um cenário empresarial realista, usando o Keycloak como componente central de identidade. A solução cobre autenticação, autorização, MFA, ciclo de vida de identidades e auditoria operacional.

O caso de uso é uma empresa fictícia com três perfis de acesso diferenciados — admin, colaborador e visitante — com uma API protegida (FastAPI) e um dashboard de demonstração interativo.

2. Arquitetura da Solução

A solução é composta por três serviços containerizados que comunicam numa rede Docker interna. O Keycloak é o único componente com conhecimento completo dos utilizadores; a FastAPI apenas valida tokens e nunca armazena estado de sessão.

Componentes

ServiçoPorta externaResponsabilidade
Keycloak 268081IdP/SSO, realm, roles, MFA, eventos
FastAPI8000API protegida, RBAC, JML proxy, dashboard
PostgreSQL 16—Persistência interna do Keycloak

Princípios de design

  • JWT validado localmente (sem round-trip ao Keycloak por pedido)
  • Admin API do Keycloak usada apenas em operações JML e auditoria
  • Configuração 100% via variáveis de ambiente (.env)
  • PostgreSQL não exposto externamente
  • Health checks encadeados garantem ordem de arranque

Separação de responsabilidades

app/auth.py Validação RS256 via JWKS, require_role() e require_mfa() como FastAPI dependencies.
app/config.py Pydantic Settings — deriva todas as URLs a partir de KEYCLOAK_URL e KEYCLOAK_REALM.
app/keycloak_client.py Helpers async para Admin REST API (uso exclusivo dentro da rede Docker).
jml/_keycloak_client.py Equivalente síncrono para scripts CLI locais (localhost:8081).

3. Decisões Técnicas

Cada escolha tecnológica foi ponderada face às alternativas, privilegiando a cobertura dos requisitos do enunciado, a reprodutibilidade e a clareza de demonstração.

3.1 Keycloak como IdP central

Vantagens
  • OIDC/OAuth2 nativo, sem código extra
  • Admin REST API completa para JML
  • Suporte a TOTP, browser flows configuráveis
  • Event logging integrado (login, MFA, admin)
  • Export/import de realm — reproduzível
Desvantagens
  • Peso em memória (~512 MB) para ambiente de desenvolvimento
  • Startup lento (~90 s) dificulta iteração rápida
  • Curva de aprendizagem nos flows de autenticação
  • Configuração UI-first — difícil de versionar sem export
Alternativa considerada: Auth0 ou Azure AD (SaaS) simplificariam o setup mas perderiam o controlo sobre o realm e a Admin API necessária para o JML. Keycloak self-hosted foi a escolha correta para o âmbito do TP.

3.2 FastAPI + JWT RS256 validado localmente

Vantagens
  • Validação sem round-trip ao Keycloak — latência mínima
  • JWKS cacheado: resiliente a falhas momentâneas do Keycloak
  • Algoritmo RS256: chave privada nunca sai do Keycloak
  • Audience e issuer validados — resistente a token replay
Desvantagens / limitações
  • Cache JWKS em memória sem TTL — requer restart após rotação de chaves
  • Revogação de token individual não suportada (JWT stateless)
  • A sessão só é cortada quando o token expira (mitigado pelo Mover/Leaver)

3.3 Clientes HTTP: async (FastAPI) vs sync (CLI JML)

httpx.AsyncClient — FastAPI
  • Não bloqueia o event loop de Python
  • Múltiplos pedidos ao Keycloak em paralelo (ex.: audit)
  • Padrão recomendado em endpoints FastAPI async
httpx síncrono — CLI JML
  • Scripts de linha de comando não precisam de event loop
  • Código mais simples e linear para operações sequenciais
  • Mesma biblioteca (httpx) — API consistente

3.4 O que poderia ter sido feito de forma diferente

Keycloak Authorization Services Policies e permissions declarativas no Keycloak (bónus do enunciado). Não implementado — RBAC no código cobre os requisitos mínimos e é mais demonstrável ao vivo.
JWKS com TTL/invalidação Cache com expiração automática de 5–15 min eliminaria a necessidade de restart manual após rotação de chaves. Deixado como melhoria futura.
Authorization Code + PKCE em demo O fluxo browser recomendado. Implementado no realm mas a demo usa ROPC para facilitar testes via curl/Swagger. Em produção, ROPC seria desativado.

4. Autenticação OIDC e Fluxo de JWT

A autenticação é inteiramente delegada ao Keycloak. A FastAPI nunca verifica credenciais diretamente — apenas valida o JWT emitido e extrai as claims necessárias.

Fluxo de autenticação (demo — ROPC)

1
Cliente envia POST /auth/token com username + password (+ totp opcional) para a FastAPI.
2
FastAPI encaminha para o token endpoint do Keycloak (http://keycloak:8080/realms/iam-tp/protocol/openid-connect/token) via grant Resource Owner Password Credentials.
3
Keycloak autentica o utilizador, aplica o flow direct grant with otp (TOTP opcional) e devolve um JWT RS256 assinado com a chave privada do realm.
4
FastAPI devolve o access_token ao cliente. O token contém realm_access.roles, amr, sub, email e preferred_username.
5
Em cada pedido subsequente, auth.py busca a JWKS (cache), resolve o JWK pelo kid do header, valida assinatura RS256, audience e issuer — sem contactar o Keycloak.

Claims utilizadas

ClaimOnde é usadaExemplo
sub/me, identificação de utilizador"abc123-..."
emailPerfil de sessão no dashboard"colab@empresa.pt"
preferred_usernameNome visual na UI"colaborador.user"
realm_access.rolesBase de todo o RBAC["colaborador"]
amrrequire_mfa() — acesso a /admin/mfa-area["otp"]
acrFallback na verificação de MFA"1"
expCountdown de expiração no dashboard1748000000
Segurança da validação: a JWKS é obtida de http://keycloak:8080/realms/iam-tp/protocol/openid-connect/certs na primeira validação e guardada em _jwks_cache. Audience (fastapi-client) e issuer (http://localhost:8080/realms/iam-tp) são validados em cada pedido, prevenindo tokens de outros realms ou clientes.

5. Autorização RBAC

A autorização é implementada como FastAPI dependencies — require_role() e require_mfa() — que injetam o utilizador autenticado e verificam as suas roles sem qualquer chamada ao Keycloak. A lógica vive inteiramente em app/auth.py.

Hierarquia de acesso

EndpointProteçãoVisitanteColaboradorAdmin
/public, /healthNenhuma200200200
/meJWT válido200200200
/colaborador/*colaborador ou admin403200200
/admin/users, /auditadmin403403200
/admin/mfa-areaadmin + MFA403403200*

* Admin necessita de MFA ativo no JWT (claim amr: ["otp"]). Sem MFA, mesmo admin recebe 403.

Vantagens do RBAC no backend

  • Roles extraídas do JWT — sem latência adicional
  • Dependency injection do FastAPI garante que o guard não é esquecido
  • Mensagem de erro explícita com roles necessárias vs. roles do utilizador
  • Testável de forma isolada (sem Keycloak nos testes)

Limitações / alternativas

  • Roles hardcoded nos endpoints — adicionar nova role requer código
  • Alternativa: Keycloak Authorization Services permite definir policies/permissions declarativamente no realm, sem tocar no código da API
  • Alternativa: OPA (Open Policy Agent) para políticas centralizadas e auditáveis
Grupos Keycloak: os grupos Admins, Colaboradores e Visitantes espelham as roles e têm role assignments automáticos. Isto permite que as operações JML de adição a grupo sejam suficientes para conceder o acesso correspondente, mantendo consistência entre o diretório e a autorização.

6. Ciclo de Vida JML e MFA

6.1 Fluxos Joiner / Mover / Leaver

Os três fluxos JML são implementados sobre a Keycloak Admin REST API. Cada operação garante atomicidade entre role e grupo, e o Mover/Leaver revogam sessões imediatamente — sem esperar pela expiração natural do token (300 s por defeito).

FluxoOperações Admin APIEfeito de segurança
Joiner POST /users → POST /role-mappings → PUT /groups/{id} Acesso inicial controlado pela role atribuída
Mover DELETE role antiga → POST role nova → DELETE grupo antigo → PUT grupo novo → DELETE /sessions Impacto imediato: próximo token já tem a nova role
Leaver PUT enabled=false → DELETE todas as roles → POST /logout Login negado, sessões ativas cortadas imediatamente
Vantagens da abordagem
  • Identidade gerida no diretório central (Keycloak), não em ficheiros locais
  • Revogação imediata de sessões após Mover/Leaver
  • Dual interface: CLI Python para operações manuais + endpoints REST para integração
  • Leaver preserva o utilizador (auditoria histórica) — apenas desativa e remove roles
Limitações
  • Mapeamento role→grupo duplicado em 3 ficheiros (admin.py, joiner.py, mover.py)
  • Sem transação — falha a meio deixa estado parcial
  • Alternativa: Keycloak Events ou Webhook para JML event-driven

6.2 MFA — TOTP via Keycloak

O realm usa o flow browser-mfa com OTP condicional: o passo TOTP só é apresentado ao utilizador se tiver uma app autenticadora configurada (conditional-user-configured). O admin.user tem CONFIGURE_TOTP como required action, o que obriga à configuração no primeiro login browser.

Fluxo de setup (primeira vez)

1
Login de admin.user → Keycloak deteta required action CONFIGURE_TOTP
2
Keycloak apresenta QR code → utilizador regista no Google Authenticator / Authy
3
Utilizador introduz código TOTP → Keycloak confirma e emite JWT com amr: ["otp"]
4
require_mfa() aceita o token → /admin/mfa-area devolve 200

Verificação na API

Sem MFA — amr: ["pwd"] require_mfa() verifica: "otp" in amr → false → HTTP 403 Forbidden
Com MFA — amr: ["otp"] require_mfa() verifica: "otp" in amr → true → utilizador passado ao handler
Nota: o claim amr é emitido pelo Keycloak com base no flow de autenticação executado. A API não consulta o Keycloak — apenas lê o claim do JWT já validado.

7. Auditoria, Riscos e Conclusão

7.1 Auditoria

O Keycloak regista eventos de utilizador e eventos administrativos no PostgreSQL — persistem entre restarts. A aplicação expõe dois endpoints de auditoria protegidos por role admin:

CategoriaEventos registados
Login / sessãoLOGIN, LOGIN_ERROR, LOGOUT
TokenREFRESH_TOKEN, CODE_TO_TOKEN
MFAUPDATE_TOTP, UPDATE_TOTP_ERROR, REMOVE_TOTP
Admin eventsAlterações de user, role, group (com detalhes)
GET /admin/audit Devolve user_events + admin_events em bruto. Suporta filtro por event_type e paginação (max_events ≤ 200).
GET /admin/audit/summary Agrupa os eventos por tipo com contagem. Útil para evidência no relatório: LOGIN: 12, LOGOUT: 5, LOGIN_ERROR: 2, UPDATE_TOTP: 1.

7.3 Conclusão

A solução cobre todos os requisitos obrigatórios do enunciado (secções 3.1 a 3.7) e os requisitos não funcionais (README reproduzível, sem secrets no código, organização clara). Os testes automatizados validam a lógica de JWT, roles e MFA sem serviços em execução.

Pontos fortes: reprodutibilidade total com docker compose up --build, dual interface JML (CLI + API), revogação de sessões em tempo real, auditoria nativa do Keycloak, e dashboard interativo de demonstração.
Melhorias futuras: JWKS cache com TTL, Keycloak Authorization Services para políticas declarativas, ROPC desativado em produção (Authorization Code + PKCE apenas), transações em JML para rollback em caso de falha.
Demo: o dashboard em http://localhost:8000/dashboard permite demonstrar todos os requisitos de forma sequencial em ~10 minutos: login → claims → RBAC → MFA → JML (Joiner/Mover/Leaver) → auditoria.
